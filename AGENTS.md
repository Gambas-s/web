# AGENTS.md

> 이 레포에서 작업하는 모든 AI 에이전트(Claude, Gemini 등)가 **작업 시작 전 반드시 읽어야 하는** 최상위 규칙입니다.
> 이 파일 하나가 워크플로우/협업레벨/브랜치/리뷰 규칙을 모두 포괄합니다.
---

## 절대 금지 사항

AI 에이전트는 다음을 **절대 하지 않는다**:

- `main` 에 직접 푸시
- 사람 승인 없이 PR 머지
- 테스트를 삭제/스킵해서 통과시키기
- 이슈 없이 큰 변경 진행
- L1/L2 결정을 단독으로 확정
- 보안 키/환경변수/자격증명을 코드에 하드코딩
- 이 문서(AGENTS.md)를 단독 수정 (§14 참조)
- GitHub 이슈 본문을 AI가 직접 수정 (제안만 가능, §10 참조)
- 이 문서 자체의 변경은 **L1 (사람이 결정)**
- 해당문서 직접 수정

## AGENTS.md 문서 규칙 
- 해당 문서는 절대 승인 없이 수정하지 않는다.
- 수정 시 반드시 `docs/decisions/` 에 변경 이유를 ADR로 남김
- 이 문서는 **쓰면서 진화**시키는 것이 원칙 — 현실과 맞지 않으면 즉시 업데이트

## 협업 철학

- **1인 개발자(Owner) + AI 에이전트들** 구조
- Owner = 방향 결정자 & 최종 승인자
- Claude = 설계/구현/셀프리뷰/문서화 주도
- 외부 AI (Gemini 등) = 2차 리뷰어 (편향 보완)
- **품질과 문서화 > 속도**

---

## 협업 레벨 L1~L4

모든 작업은 시작 전 레벨을 선언한다. 애매하면 **더 높은 레벨**(사람 개입이 많은 쪽)을 따른다.

| 레벨 | 주도 | 승인 | 해당 작업 |
|------|------|------|---------|
| **L1** | 사람 | — | 제품 방향, 비즈니스 결정, 릴리스, 머지, 이 문서 수정 |
| **L2** | AI 제안 → 사람 결정 | 사람 | 아키텍처, DB 스키마, API 설계, 작업 분해 |
| **L3** | AI | 사람 + 외부 AI | 이슈 작성, 구현, 리팩터링, 버그 픽스 |
| **L4** | AI | 자동 | 테스트, 문서, 포맷팅, 타입힌트, 오타 수정 |

## 매크로 / 마이크로 사이클

개발은 **2단 구조**:

### 매크로 사이클 (이슈 단위)

> 💡 **각 단계마다 이슈 본문의 "📊 진행 추적" 섹션을 업데이트한다** (§10 참조).

```
[1] 요구사항 정리       (L1)
    ↓
[2] 이슈 작성           (L3) — .github/ISSUE_TEMPLATE/task.md 사용
    ↓
[3] 설계 (필요시)       (L2) — docs/decisions/NNNN-*.md 작성
    ↓
[4] 브랜치 생성         (L3) — develop에서 분기
    ↓
[5] 작업 분해           (L2) — 마이크로 사이클 리스트를 제시하고 사람 승인
    ↓
[6] 마이크로 사이클 반복 (아래 참조)
    ↓
[7] 통합 셀프 리뷰      (L4) — 통합 셀프 리뷰 체크
    ↓
[8] 피드백 반영         (L3)
    ↓
[9] 사람, Gemini 최종 리뷰     (L1)
    ↓
[10] PR 머지 → develop  (L1) — Squash and Merge

```

### 통합 셀프 리뷰 체크

모든 MC 완료 후 이슈 전체 관점:

### 기능
- [ ] 이슈의 모든 수용 기준 충족
- [ ] 전체 테스트 통과 (커버리지 확인)
- [ ] 엣지 케이스 커버 (null, 빈값, 음수, 경계값, 동시성)

### 품질
- [ ] 중복 제거, 단일 책임 원칙
- [ ] MC들이 합쳐진 상태에서 일관성
- [ ] 린트/포맷 통과

### 보안
- [ ] 입력 검증, 인젝션 방어
- [ ] 인증/인가 체크
- [ ] 비밀키/토큰 노출 없음
- [ ] 에러 메시지에 민감 정보 누출 없음

### 성능
- [ ] N+1 쿼리 없음
- [ ] 불필요한 반복/재계산 없음

### 에러 처리
- [ ] 예상 에러 모두 처리
- [ ] 실패 시 부분 상태 남지 않음 (트랜잭션/롤백)

### 문서
- [ ] 공개 API docstring
- [ ] README 최신
- [ ] 복잡 로직 주석
- [ ] CHANGELOG (릴리스 예정시)

### 마이크로 사이클 (작업 단위)

```
🔴 RED → 🟢 GREEN → 🔵 REFACTOR → 🤖 미니 셀프 리뷰(5개) → 📝 커밋 → ✋ 보고+승인 대기
                                                                         ↑
                                                                         └── 다음 MC로 루프
```

**작업 분해 단위 기준**
- ✅ 30분~2시간 내 완료
- ✅ 독립적으로 테스트 가능
- ✅ 커밋 1개로 의미 있음
- ❌ 너무 작음 (변수명 변경 수준) / 너무 큼 (여러 파일 대대적 수정)

**각 MC 종료 시 Claude의 보고 형식**
```
✅ MC-N 완료: [제목]

변경: [파일들]
미니 셀프 리뷰: [5개 체크 결과]
커밋: [해시 + 메시지]
의문점/제안: [있으면]

다음: MC-(N+1) [제목] 진행할까?
```

## 4. 체크인 모드

현재 기본값: **🔴 타이트**

| 모드 | 체크인 빈도 |
|------|----------|
| **🔴 타이트 (기본)** | 매 MC 종료 시 보고 + 승인 대기 |
| 🟡 미디엄 | 작업 분해만 승인, MC는 보고만 |
| 🟢 자율 | 전체 이슈 완료 후 한 번에 보고 |

이슈 단위로 모드 변경 가능 (예: "이건 미디엄으로").
모드 명시 없으면 타이트.

## TDD 원칙

- 새 기능/수정은 **실패하는 테스트부터**
- 🔴 RED → 🟢 GREEN → 🔵 REFACTOR
- 반드시 TDD 한번에 하나의 테스트만 수행, 몇십개의 테스트를 짜고 동시에 RED GREEN을 하지 않음
- 테스트 없는 커밋은 L4(문서/포맷팅)가 아닌 이상 지양
- 테스트를 삭제/스킵해서 통과시키기 **금지**

## 브랜치 & 커밋 규칙

### 브랜치 (Git Flow)

| 브랜치 | 용도 | 베이스 | 머지 대상 |
|--------|------|-------|----------|
| `main` | 프로덕션 | — | `release/*`, `hotfix/*` 만 |
| `develop` | 개발 통합 | — | feature/fix/refactor |
| `feature/{이슈번호}-{슬러그}` | 기능 | `develop` | `develop` |
| `fix/{이슈번호}-{슬러그}` | 버그 | `develop` | `develop` |
| `refactor/{이슈번호}-{슬러그}` | 리팩터링 | `develop` | `develop` |
| `release/v{X.Y.Z}` | 릴리스 준비 | `develop` | `main` + `develop` |
| `hotfix/v{X.Y.Z}-{슬러그}` | 긴급 수정 | `main` | `main` + `develop` |

슬러그: 영문 소문자 + 하이픈, 3~5 단어.

### 커밋 메시지 (Conventional Commits)

```
<type>(<scope>): <subject> (#이슈번호)
```

타입: `feat` | `fix` | `refactor` | `test` | `docs` | `style` | `perf` | `chore` | `ci`

예시:
```
feat(auth): add OAuth login flow (#42)
test(auth): add failing tests for token validation (#42)
refactor(auth): extract token validator (#42)
```

## 이슈 본문 추적 프로토콜

> **핵심 원칙**: GitHub 이슈 본문은 "살아있는 문서"다. 작업 진행에 따라 계속 업데이트된다.
> Claude는 세션 간 메모리가 없으므로, **이슈 본문이 진실의 원천(Source of Truth)** 이다.

### 이슈 본문의 "📊 진행 추적" 섹션

이슈 템플릿(`task.md`)의 "진행 추적" 섹션은 다음을 포함한다:

- **🏷️ 상태**: 현재 단계 / 브랜치 / PR 번호
- **📦 작업 분해**: MC 체크리스트 + 커밋 해시
- **🔍 리뷰 체인**: 셀프/외부/사람 리뷰 진행
- **📝 주요 결정사항**: L2 결정 요약 (상세는 ADR)

### Claude의 업데이트 제안 의무

Claude는 다음 시점마다 **이슈 본문 업데이트를 사람에게 제안**한다:

| 시점 | 업데이트 항목 |
|------|-------------|
| 작업 분해 승인 직후 | 상태 `📋 분해완료`, MC 리스트 채우기 |
| 브랜치 생성 직후 | 상태 `🔨 개발중`, 브랜치명 |
| 각 MC 완료 시 | 해당 MC 체크박스 ✅ + 커밋 해시 |
| L2 결정 발생 시 | 주요 결정사항 요약 + ADR 링크 |
| 통합 셀프 리뷰 완료 | 리뷰 체인 첫 박스 ✅, PR 번호 기입 |
| 외부 AI 리뷰 완료 | 리뷰 체인 두 번째 박스 ✅ |
| 사람 리뷰 완료 | 리뷰 체인 세 번째 박스 ✅, 상태 `✅ 완료` |


## 결정 기록 (DECISIONS)

중요한 기술 결정은 `docs/decisions/NNNN-slug.md` 에 기록한다.

- **NNNN**: 4자리 순번 (0001부터)
- **템플릿**: `docs/decisions/0000-template.md`
- **언제 기록**: 아키텍처, DB, 주요 라이브러리 선정, API 설계 방식, 보안 정책 등
- **언제 스킵**: 단순 구현 선택, 버그 픽스 방법

**유연한 진화 원칙**
- 기존 결정이 바뀌면 → 새 ADR 작성 + 이전 ADR 상태를 `Superseded by ADR-NNNN` 으로 변경
- 이 파일(AGENTS.md) 자체의 규칙이 바뀌면 → DECISIONS에 "왜 바꿨는지" 기록

   - 주요 결정사항 / ADR 링크
